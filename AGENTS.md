# AGENTS.md

这是 `115m` 项目的 AI 协作规则。

## 项目定位

- 项目名：`115m`
- 类型：Chrome 扩展
- 包管理：统一使用 `pnpm`

## 协作分工

- 用户负责：提需求、测试、反馈结果
- AI 负责：看代码、改代码、跑命令、处理细节、汇报结果
- 能直接做的事，AI 直接做，尽量不把零碎操作再丢给用户

## 文档原则

- `AGENTS.md` 只写核心规则、默认动作、文档路由，不堆实现细节
- `docs/AI-PREFERENCES.md` 只写稳定、长期有效的用户偏好，保持很短
- `README.md` 只写项目说明、目录、命令、使用方式
- `项目日志.md` 只记关键版本、重要重构、仍会影响后续决策的变化
- `docs/AI-HANDOFF.md` 只写项目地图、模块职责、常见落点、接手说明
- `docs/runbooks/*.md` 视为项目内 skills，只写专题流程，按需查看
- 文档不要重复堆同一类内容

## 任务路由

- 涉及发布、文档整理、输出风格、协作方式：先看 `docs/AI-PREFERENCES.md`
- 涉及播放器核心、平台层、消息链路：先看 `docs/AI-HANDOFF.md`
- 涉及图片图墙、文件夹封面、图片查看器、混合目录展示策略：先看 `docs/runbooks/image-wall.md`
- 涉及 GitHub Release、版本打包、发布说明：先看 `docs/runbooks/release.md`
- 只做普通代码修改时：先看本文件，再直接看相关代码，不默认通读全部文档
- 发现某类任务反复出现且步骤固定时，应补一份新的 `docs/runbooks/*.md`

## Skill 机制

- `docs/runbooks/*.md` 视为项目内 skills，AI 应按任务类型按需查看，不默认全部阅读
- skill 只解决一类重复任务，保持短、可执行、可更新，不写大段背景说明
- 同类任务多次重复且步骤已经稳定时，AI 应主动收敛到现有 runbook，或补一份新的 runbook

## 偏好记忆规则

- 用户明确表达且长期有效的偏好，AI 应沉淀到仓库文档，而不是只依赖会话记忆
- 项目级偏好写入 `AGENTS.md`
- 协作风格、输出习惯、发版口径这类稳定偏好写入 `docs/AI-PREFERENCES.md`
- 某类任务的固定做法和检查步骤写入对应 `docs/runbooks/*.md`
- 更新偏好时优先覆盖旧表述和压缩内容，不无限追加，避免记忆文档膨胀
- 目标是让后续 AI 用更少上下文完成任务，而不是让每次任务都预读全部文档

## 发布说明规则

- 发布 GitHub Release 时，默认写面向用户的更新说明，不写内部技术细节、提交名堆砌或纯底层重构内容
- 发布说明优先总结用户可直接感知的变化，例如修了什么体验问题、哪些地方更快、更稳、更顺手
- 发布说明内容优先参考 `项目日志.md`，如果不够，再结合本次版本区间提交补全用户可感知的变化

## 主线保护

- `main` 只保留已验证、可发布、可回退的稳定代码
- 非用户明确要求时，AI 不直接在 `main` 上做优化、重构、试验性改动
- 需要迭代、优化、重构、试错时，先建立独立分支，再开始修改
- 分支验证通过后，才能合并回 `main`
- 不把半成品、未验证代码、临时调试代码带进 `main`
- 如果当前就在 `main`，且任务属于优化、重构、试验，先提醒并切到新分支再继续

## 分支收口规则

- AI 要主动判断当前分支是否已经达到“可合并”状态，避免长期堆积无用分支
- 满足以下条件时，视为可以合并回 `main`：需求范围已完成、`pnpm build` 通过、需要时 `pnpm test` 通过、用户实际验证未发现明显问题、没有已知高风险问题
- 未满足以上条件时，不合并回 `main`
- 合并目标是让 `main` 保持整洁稳定，不为了清理分支而带风险合并
- 合并完成后，如无特殊原因，分支应及时收口，不长期保留无意义支线
- 未经用户明确要求，AI 不推送远程；本地是否进入可合并状态由 AI 主动判断并汇报

## 开发护栏

- 涉及播放器控制栏、覆盖层、播放记忆、播放列表、消息链路等核心结构时，先审查现有底座与官方能力，禁止凭感觉直接补丁式改 UI 或交互
- 官方库已提供稳定能力时，优先使用官方能力和官方扩展点实现；不能先绕开官方机制做漂浮层、绝对定位补丁，后面再返工
- 遇到结构性问题时，先判断是配置问题、扩展点问题还是底座设计问题；如果底座不对，先整理底座，再继续叠加功能
- 新功能实现要默认考虑完整链路：初始化、正常使用、切换/跳转、结束态、异常态、销毁与状态清理，不能只让单一路径“看起来能用”
- 对同一问题连续两次修补仍不稳定时，停止继续打补丁，改为回到底层重新设计实现路径
- 涉及播放器核心、`115` 页面能力、消息链路时，按 `docs/AI-HANDOFF.md` 的模块边界落代码，不回填大文件
- 后续新增功能时，先复用现有模块，不把新逻辑继续堆进超大文件
- URL 参数解析、播放列表整理、播放状态决策、接口访问，优先拆到独立小模块
- 新增或修改 `chrome.runtime` 消息时，要同步更新 `src/shared/messages.ts`，不能裸写消息结构
- 同一类降级策略、鉴权策略、下载策略、播放源策略不得在多个业务文件重复实现，优先抽到共享模块统一维护
- 能写成纯函数的逻辑，优先写成纯函数，并补最小必要测试
- 改动播放器核心链路时，优先保证：打开播放、画质切换、播放记忆、移动文件 这四条主链路不回退
- 当主链路稳定、测试通过、用户实测正常后，AI 不为追求形式上的“更干净”继续大规模重构；后续以真实需求和真实问题驱动调整

## 修改后的默认动作

- 改了代码、功能、配置、打包相关内容后，默认执行 `pnpm build`
- 改到逻辑、状态、消息、播放器核心链路时，默认同时执行 `pnpm test`
- 架构调整、播放器核心改动、消息链路改动，在 `pnpm build`、`pnpm test` 通过后，仍需用户实际验证主链路正常，才进入合并 `main` 判断
- 涉及代码、功能、架构、规则的重要变更后，默认同步更新 `项目日志.md`
- 只改文档、规则、日志时，不用跑构建
- 只改文案、注释、纯样式时，可以不跑 `pnpm test`
- 除非用户明确说不用构建，否则代码改动后不能跳过构建
- 构建结果要用中文汇报

## 构建汇报格式

- 构建结果：成功 / 失败
- 执行命令：`pnpm build`
- 如果失败：说明关键报错
- 如果失败：说明有没有继续尝试修复

## 安全边界

- 不泄露本地密钥、账号、环境信息等敏感内容
- 不做未经确认的破坏性操作
