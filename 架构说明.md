# 115m 专业化架构说明

## 目标

- 让功能迭代从“补丁叠加”转为“模块化扩展”。
- 让核心链路可定位、可维护、可验证。
- 保证开发过程统一走 `pnpm build`，避免环境分叉。

## 当前分层

### content 层

- 入口：`src/content/home.ts`
- 领域模块：
  - `src/content/core/extractors.ts`：点击目标与文件信息提取
  - `src/content/core/preview.ts`：列表预览图渲染
  - `src/content/core/player-open.ts`：打开播放器与预热触发
  - `src/content/core/runtime.ts`：runtime 消息调用封装
  - `src/content/core/utils.ts`：通用工具（时长解析、并发调度）

### player 层

- 入口：`src/player/player.ts`
- 领域模块：
  - `src/player/core/source.ts`：视频源获取与重试策略
  - `src/player/core/quality.ts`：画质选项构建与映射
  - `src/player/core/history.ts`：播放历史读写
  - `src/player/core/runtime.ts`：runtime 消息调用封装
  - `src/player/core/types.ts`：播放器核心类型

### background 层

- 入口：`src/background/index.ts`
- 职责：消息路由、预热缓存、下载/历史持久化。

### shared 层

- `src/shared/messages.ts`：content/player/background 共用消息协议类型。

## 已落地规范

- 命名统一：项目名与日志前缀统一为 `115m`。
- 构建规范：开发改动后统一执行 `pnpm build` 验证。
- 发布规范：`pnpm zip` 仅用于手动打包发布。
- 仓库纯净：`release/` 整体忽略，不再跟踪发布产物。

## 后续开发原则

- 新功能先定边界（content/player/background/shared），再落代码。
- 不在入口文件中堆叠业务细节，优先新增领域模块。
- 所有跨层通信先更新 `src/shared/messages.ts`。
- 每次改动至少通过一次 `pnpm build`。

## 开发流程（固定）

- 开发验证：`pnpm build`
- 人工冒烟：按 `开发验收清单.md` 执行
- 发布打包：`pnpm zip`

## 技术债状态

- 已清理的技术债：
  - 历史旧命名混用（115Master/115master）
  - content 与 player 入口文件职责过重
  - 消息协议分散定义
  - 发布产物被仓库跟踪
- 仍需持续治理（非阻塞）：
  - `player` 打包体积较大（仅警告，不影响功能）
  - 缺少自动化测试（当前靠构建+手工回归）

结论：当前架构已从“补丁式”进入“可持续扩展”状态，可继续在该架构上稳定迭代。
