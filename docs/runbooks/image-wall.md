# Image Wall Runbook

适用场景：

- 设计或实现图片图墙视图
- 设计或实现文件夹封面卡片
- 设计或实现图片查看器替换
- 处理混合目录里的图片、文件夹、视频共存展示

## 当前方向

- 图片目录能力按“自定义图墙视图”设计，不再按原生列表模式补一条小预览
- 图片卡片和文件夹卡片共用网格，但文件夹必须保留明确的文件夹视觉语言
- 旧 `115魔改` 只作为参考，不直接整包搬进当前项目
- 这类能力默认先在独立分支实验，不直接进 `main`

## 已确认数据来源

- 图片文件在列表模式下可直接从列表项读取缩略图：优先 `path`，兜底 `img[src]`
- 文件夹在列表模式下虽然原生不显示封面，但列表项已有 `img_url`
- 文件夹封面在缩略图模式下显示的也是同一个 `img_url` 来源，不是另一套独立接口

## 视觉原则

- 目标是“一眼扫很多图”，不是一行一个文件的列表补丁
- 横图、竖图都必须完整显示，默认 `object-fit: contain`
- 不把 115 提供的小缩略图硬拉太大，观感主要靠卡片容器设计，不靠强放大图片
- 图片卡片保持干净；文件夹卡片必须有 folder tab、边框或标题区等明显文件夹标识
- 文件夹封面优先使用用户设置封面；没有时再退回自动封面或默认文件夹样式

## 稳定性原则

- 不影响当前视频主链路：打开播放、画质切换、播放记忆、移动文件
- 不先改播放器核心，不先改现有消息主链路
- 优先落在 `src/content/`，作为独立视图层实现
- 原始列表数据和原始 DOM 尽量保留，可快速切回
- 新能力要可关闭，出现问题能单独隔离

## 混合目录规则

- 必须默认考虑目录内同时存在文件夹、图片、视频的情况，不能只按纯图片目录设计
- 不能和现有视频列表预览冲突，尤其不能重复接管同一个视频项
- 混合目录推荐策略：
1. 文件夹和图片进入图墙卡片
2. 视频继续走现有视频预览链路，除非后续明确要为视频设计单独卡片样式
3. 如图墙实验阶段无法把三类项目稳定混排，优先只在“图片占多数”或“明确图片目录”启用

## 实现建议

- 只读取现成字段，不为封面额外请求接口：
1. 图片：`path` / `img[src]`
2. 文件夹：`img_url`
- 使用 `IntersectionObserver` 懒加载可视区域卡片
- 控制并发和缓存，避免长列表抖动
- 点击图片时接入自定义查看器；点击文件夹时仍走原生进目录逻辑

## 开发前确认

1. 这次做的是图墙视图、图片查看器、文件夹封面，还是左侧栏
2. 是否涉及混合目录；如果涉及，视频项如何避让
3. 是做完全替换视图，还是做可切换视图
4. 当前是否在实验分支；若在 `main`，先切分支

## 验证重点

- 纯图片目录显示正常
- 带文件夹封面的目录显示正常
- 混合目录不和视频预览冲突
- 目录切换、返回上级、滚动加载、销毁重建不残留脏节点
- 不影响现有视频主链路
