# 文档与规则维护规范

适用场景：

- 整理项目文档结构时
- 判断内容应该写入哪里时
- 发现规则、偏好、流程或项目地图混写时

## 分层原则

- `AGENTS.md`：只放项目级核心总纲
- `docs/AI-PREFERENCES.md`：只放长期稳定偏好
- `docs/AI-HANDOFF.md`：只放项目地图、模块接手与需求落点
- `docs/runbooks/*.md`：只放固定流程或确实复杂、容易遗漏的步骤清单
- `.trae/rules/**/*.md`：只放给 Trae 自动生效的规则

## Trae 规则生效方式

- `alwaysApply: true`：始终生效，适合项目级铁律
- `description`：智能生效，适合某类任务场景
- `globs`：指定文件生效，适合模块或文件类型规则
- 手动触发：适合低频、边界清楚、希望由用户主动控制的规则
- 一个规则可同时使用 `description` 与 `globs`，表示既有场景语义，也有限定文件范围

## 当前项目规则设计

- `project-baseline.md`：始终生效，承载项目核心规则
- `docs-maintenance.md`：智能生效 + 指定文件生效，用于整理文档、规则、偏好与 runbook
- `release.md`：智能生效，用于发布与发布说明
- `messages.md`：智能生效 + 指定文件生效，用于消息链路相关文件与任务
- `player.md`：智能生效 + 指定文件生效，用于播放器相关文件与任务
- `git-commit-message.md`：智能生效，用于生成提交信息

## 精简结论

- 能直接由短规则说明清楚的场景，不单独再做文档路由
- `release.md` 已收缩为轻量智能规则；当前不依赖 `项目日志.md`
- `messages.md`、`player.md` 保留简短 runbook，仅承载落点索引和必要检查，不再重复规则约束
- `image-wall.md` 仍保留为独立 runbook，因为它包含较多设计约束、稳定性边界与实验策略

## 写入规则

### 写入 `AGENTS.md`

仅收录：

- 项目定位
- AI 工作方式
- 改动边界
- 隔离原则
- 协议约束
- 默认验证要求
- 文档路由
- 安全要求

不收录：

- 模块详细地图
- 某个子系统的实现细节
- 低频特殊流程
- 一次性需求

### 写入 `docs/AI-PREFERENCES.md`

仅收录：

- 长期稳定的用户偏好
- 长期稳定的产品交互约定
- 会长期影响后续实现判断的边界

不收录：

- 项目公共铁律
- 临时修复记录
- 场景流程

### 写入 `docs/AI-HANDOFF.md`

仅收录：

- 项目结构总览
- 模块职责
- 典型任务落点
- 接手顺序
- 文档路由索引

不收录：

- 长流程操作说明
- 临时需求过程
- 重复的核心规则

### 写入 `docs/runbooks/*.md`

仅收录：

- 某一类固定流程
- 确实复杂、容易遗漏的步骤清单
- 模块落点索引
- 必要检查项

推荐结构：

1. 适用场景
2. 默认原则或前提
3. 常见落点或操作步骤
4. 必要检查或验证重点

### 写入 `.trae/rules/**/*.md`

仅收录：

- 希望 Trae 自动套用的规则
- 与文件类型、任务场景或项目全局相关的规则

不收录：

- 纯说明性文档
- 大段背景介绍
- 一次性任务记录
- runbook 中的完整操作步骤与完整检查清单

## 维护原则

- 新内容先判断归属，再写入对应层
- 同一条规则不要在多个地方重复维护不同版本
- `.trae/rules/` 负责短规则；只有复杂且易漏的场景才额外保留 runbook
- 项目级铁律优先放到 `alwaysApply: true` 规则
- 场景型规则优先使用 `description`
- 文件范围明确的规则优先补 `globs`
- 文件范围较宽的规则，要补一条“不要误扩大判断范围”的约束
- 如果内容已升级为项目公共铁律，迁移到 `AGENTS.md`
- 如果内容只在特定流程下有效，迁移到对应 runbook
- 如果内容需要让 Trae 自动生效，迁移到 `.trae/rules/`
