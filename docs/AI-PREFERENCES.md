# AI-PREFERENCES

适用场景：

- 发布 GitHub Release
- 整理文档或规则
- 需要判断输出风格、协作方式时

## 长期偏好

- 默认直接动手做，少把零碎步骤再丢给用户
- 发布 GitHub Release 时，默认写给用户看的更新说明，不写内部技术细节
- 文档尽量短：核心规则留在 `AGENTS.md`，细节放到按需查看的 runbook
- `项目日志.md` 只记关键变化和仍影响后续判断的内容，早期不重要历史可省略
- AI 默认不要通读全部文档，应先看路由，再按任务加载必要文档
- 本地支线默认只做临时迭代，不主动推送；AI 负责判断何时可合并到 `main`，一旦合并到 `main` 就继续进入发布更新流程
- 对用户已明确确认、且后续容易遗忘或回退的功能偏好、交互约定、实现边界，AI 应主动实时更新到这里，不等用户提醒
- 只要是值得长期保留、会影响后续实现判断的稳定偏好，AI 应主动沉淀到这里
- 图片查看器交互约定：正常模式下滚轮切图，缩放模式下滚轮缩放，双击负责进入或退出缩放模式
- 图片查看器滚轮切图原则：应优先保证结果可预期、顺序推进、避免一次手势直接跳多张；不同设备的原始 wheel 差异应由实现层吸收，而不是直接暴露给用户

## 维护规则

- 这里只保留稳定、重复出现、长期有效的偏好
- 新偏好应优先合并或替换旧表述，不堆相似条目
- 如果某条偏好已经变成项目公共规则，应迁移到 `AGENTS.md`
